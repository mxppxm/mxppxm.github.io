# 🤖 我给自己写了个 AI 记账助手 — 从灵感到踩坑再到上线

> 一个前端工程师的 AI Skill 开发实录：如何用 Python 给自己的 AI 搭档「注入灵魂」，让它学会帮你管钱。

## 为什么会有这个想法？

事情要从某天深夜说起。

我盯着银行 App 里一笔笔消费记录，陷入了沉思——这笔 30 块的「餐饮」到底是午饭还是咖啡？这笔「交通」是打车还是地铁？更关键的是，**我明明设了每月 4000 的预算，每次都是在超支的边缘才后知后觉。**

我需要一个记账工具，但我不想下载新 App，不想注册新账号，不想每次消费完还要打开另一个界面点半天。我想要的是——**像跟朋友说「今天花了30买咖啡」一样自然地记账。**

恰好在折腾 [Clacky](https://github.com/openclacky/clacky) 这个 AI 平台，它支持自定义 Skill（技能），于是我想：**为什么不给我的 AI 搭档装个「财务大脑」呢？**

## 第一版：开干

思路很简单：解析自然语言 → 写入 CSV → 汇总统计。

```
用户说：今天支出30元买咖啡
→ AI 解析出：日期=今天、类型=支出、金额=30、类别=餐饮
→ 写入 ~/.clacky/skills/记账助手/记账本.csv
→ 回复：已记录 ✅ 本月已支出 XXX 元
```

听起来很美好，对吧？结果一上手就翻车了。

## 踩坑实录

### 🕳️ 坑一：「工资到账5000」到底算什么？

我的第一版用了三条正则依次匹配：

```python
# 🚫 错误的做法
income_patterns = [
    r'收入\s*(\d+(?:\.\d+)?)\s*元?\s*(.+)',
    r'到账\s*(\d+(?:\.\d+)?)\s*元?\s*(.+)',
    r'工资\s*(\d+(?:\.\d+)?)\s*元?\s*(.+)',
]
```

结果「工资到账5000元」一进来，第三条匹配上了，`(.+)` 把「到账」吞进了类别字段。我收到的数据变成了：

> 收入 5000元 — 类别：到账（❌）

更离谱的是「工资5000到账」——匹配后类别变成了「到账」，好好的工资记录被打上了「到账」标签。

**解决方案**：抛弃顺序匹配，改用**关键词加权法**。数一数消息里收入关键词和支出关键词各出现几次，哪个多就归哪类。同时预设类别映射，出现「工资」自动归工资类、「打车」自动归交通类。

```python
# ✅ 正确的做法
income_score = sum(1 for kw in INCOME_KEYWORDS if kw in message)
expense_score = sum(1 for kw in EXPENSE_KEYWORDS if kw in message)
type_ = '收入' if income_score >= expense_score else '支出'
```

### 🕳️ 坑二：昨天花了50和前天花了100

用户说话是有时间感的。他们会说「昨天打车花了50」「前天吃饭花了120」「3月5号交房租」。

但第一版只认 `datetime.now()`——**所有记录都被打上当天的日期。**

听起来是小问题？想想月末对账时的场景：

> 「我明明记得25号交的房租啊，怎么账本上写着29号？」

**解决方案**：加了个 `parse_date()` 函数，支持昨天/前天/N天前/月日/标准日期格式的智能解析。

```python
昨天 → 2026-06-28
前天 → 2026-06-27
3月5号 → 2026-03-05
5天前 → 2026-06-24
2024-01-15 → 2024-01-15
（不写日期）→ 今天
```

### 🕳️ 坑三：「打车」是交通还是「花了30打车」

另一个有意思的问题：用户说「打车30元」，金额后面没有任何文字了，那备注怎么办？

一开始备注是空的，用户一脸问号：「我花了30块，但备注是空的？」

**解决方案**：如果金额后面没文字，就把整条消息当备注。保留了原始语境，对账时一目了然。

### 🕳️ 坑四：预算的「跨月革命」

设「下月预算 5000元」这个需求听起来简单，但实现起来有点绕：

- 1月设了预算 4000
- 1月又设了「下月预算 5000」
- 2月来的时候，系统应该自动用 5000
- 3月呢？回到 4000 还是保持 5000？

**解决方案**：在预算 JSON 文件里存一个 `budget_month` 字段，跨月自动应用 `next_month_budget`，并把它提升为新的 `monthly_budget`。这样用户只需要设一次下月预算，后续自动接管。

### 🕳️ 坑五：0.1 + 0.2 ≠ 0.3

经典的浮点数陷阱。如果用 `float` 记账，一个月下来可能有几分钱的对不上。

**解决方案**：全部改用 Python 的 `Decimal` 类型，金额存储为 `"30.00"` 字符串，计算时用 `Decimal` 保证精度。

## 最终成果

折腾了一圈，最终得到的记账助手长这样：

### 📝 怎么用

| 你说 | 效果 |
|------|------|
| `今天支出30元买咖啡` | → 餐饮 30元 ✅ |
| `工资到账5000` | → 工资 5000元 ✅ |
| `昨天打车花了50元` | → 交通 50元，日期自动为昨天 ✅ |
| `预算 5000元` | → 本月预算设为 5000 ✅ |
| `预算 6000元 下个月` | → 下月自动切换 ✅ |

### 🧠 技术架构

```
用户消息 → parse_message()
           ├── 关键词加权 → 判断收支类型
           ├── 提取金额（Decimal 精度）
           ├── parse_date() → 解析日期
           └── 关键词匹配 → 自动归类

记账本（CSV） ← add_entry()
     ↓
月度统计 → 收入/支出/结余
     ↓
预算检查 → 70%提醒 / 90%告警 / 超支告警
     ↓
结构化响应 → handle() 返回 dict
```

### ⚙️ 一些设计决策

**为什么用 CSV 而不是 SQLite？**

因为我需要**人类可读**。CSV 可以用 Excel 打开、可以 `cat` 查看、可以用 `grep` 搜索。对于个人记账来说，简单就是最好的。

**7 类支出 + 4 类收入**

| 支出 | 关键词举例 |
|------|-----------|
| 餐饮 | 吃、饭、咖啡、奶茶、外卖、火锅… |
| 交通 | 打车、地铁、公交、滴滴、加油… |
| 购物 | 衣服、淘宝、京东、超市… |
| 娱乐 | 电影、游戏、KTV、旅游… |
| 医疗 | 看病、药、医院、体检… |
| 学习 | 书、课程、培训… |
| 居住 | 房租、水电、物业、网费… |

| 收入 | 关键词举例 |
|------|-----------|
| 工资 | 工资、薪水、月薪 |
| 奖金 | 奖金、年终、绩效 |
| 兼职 | 兼职、接单、外包 |
| 投资 | 理财、基金、股票、利息 |

## 一点感悟

写这个 Skill 的过程，其实就是一个**把模糊需求翻译成精确逻辑**的过程。

用户说「花了30」，背后藏着「支出」「今天」「30元」「不确定什么类别」四层信息。每一步解析都是一个小选择——猜对了用户爽，猜错了用户骂。

但正是这些「猜」的过程，让我重新思考了一个问题：**好的 AI 工具，不是让用户去适应机器的规则，而是让机器去适应人类的语言。**

记账这个场景尤其如此。没人会在消费完的那一刻说「我要发起一笔金额为30.00元、类型为支出、类别为餐饮的交易」。人们说的是「哦对，今天买咖啡花了30」。

**把「人话」翻译成「机话」，就是我这个 Skill 存在的意义。**
